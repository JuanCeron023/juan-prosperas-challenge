Operating Status Tracking for Rules Engine Actuals

1. Context

Currently, operating status tracking only triggers from the manual actual schedule flow (ActualScheduleService.Create/Update/SaveActualSchedule). When the rules engine generates actuals via generate-schedules, no operating status events are produced. This means entities with rules-engine-generated actuals do not get automatic operating status updates.

This document analyzes two proposals to integrate rules engine actuals with operating status tracking, including edge cases and tradeoffs.

2. How the Current Manual Flow Works

2.1 The Routing Decision: DefineTracker

When a manual actual schedule is saved, OperatingStatusTrackSchedule generates FutureEvents for each slot (one for openTime, one for closeTime). DefineTracker compares each event's triggerTime against the current entity time to decide where to send it:

TimingDestinationWhat Happens
Future (triggerTime > entityTime)SQS onlyfuture-event-go stores as NOT_SENT in FutureEvents collection. Poller sends to mono when time arrives.
Same minute (triggerTime ≈ entityTime)Kinesis + SQS(triggerTime=0)Kinesis updates status immediately via mono. SQS neutralizes any existing FutureEvent.
Past, same day (triggerTime < entityTime, same business day)Kinesis + SQS(triggerTime=0) if conditions metSame as above, but only if shouldProcessPastEvent returns true.
Past, different dayDiscardedNo longer relevant.

2.2 Why triggerTime=0 Exists

The schedules monorepo does not have direct access to the Operations database where FutureEvents are stored. It can only communicate through SQS (to future-event-go) and Kinesis (to the mono).

When a slot that was in the future gets moved to the present/past, there's an orphaned FutureEvent in MongoDB with the old triggerTime and status NOT_SENT. If left alone, the poller would send it at the old time, causing an incorrect status change.

Since schedules can't delete the FutureEvent directly, it sends an SQS message with triggerTime=0 (Go zero value: 0001-01-01T00:00:00Z). future-event-go finds the existing FutureEvent by sourceID + isOpenTime + NOT_SENT and updates its triggerTime to year 0001. The poller eventually picks it up, but the mono discards it because the 27-hour check fails: math.Abs(year0001 - now) >> 27 hours → falls to default case → skip.

Example: Entity has a slot opening at 5PM (future). FutureEvent exists: {sourceID: slot.ID, isOpen: true, triggerTime: 5PM, NOT_SENT}. User moves open to 10AM (past, it's 1PM now). Without triggerTime=0, the 5PM FutureEvent would fire and set OPERATING at 5PM incorrectly. With triggerTime=0, the FutureEvent gets neutralized.

2.3 The shouldProcessPastEvent Logic

For past events on the same day, DefineTracker uses shouldProcessPastEvent which checks:

  1. Is the current actual in range now? (entity is currently operating in this slot)
  2. Was the old actual in range or before the current time? (slot was moved from future to past)
  3. Is it a REFURB closed reason? (always process)

This requires access to both the current and old actual schedules.

2.4 The sourceID Problem

future-event-go deduplicates FutureEvents by sourceID + eventType + isOpenTime + NOT_SENT. The sourceID is the slot's MongoDB ObjectID. In manual schedules, this ID is stable across updates. In the rules engine, slot IDs change on every regeneration because deprecated schedules are replaced with new documents in ScheduleCache.

Solution for both proposals: Use scheduleID instead of sourceID as the deduplication key. The scheduleID is stable in both flows:

  • Manual actuals: UUID generated once on create, preserved on updates via findExistingScheduleID
  • Rules engine: Deterministic hash: MD5(businessDate + dayConfig.slotId + scheduleType) + "_actual" — does not depend on open/close times, confirmed by TestGeneratedScheduleID_Deterministic and TestGeneratedScheduleID_StableAfterRuleChanges

3. Proposal A: New Lambda + Smart Processing in future-event-go

3.1 Architecture

Schedule Changes Stream (Kinesis)
    → New Lambda (consumer)
        → GET v2 to fetch current actuals
        → Filter only scheduleSource: RULES
        → Send ALL engine slots as a batch SQS message to future-event-go
          (no DefineTracker logic in the lambda — just pass the data)

SQS → future-event-go
    → NEW "full sync" processor:
        1. Receive batch for entity + businessDate
        2. Read all existing engine FutureEvents NOT_SENT for this entity + businessDate
        3. Compare existing vs incoming per scheduleID
        4. Apply smart logic per slot (see decision tree below)

3.2 The Decision Tree in future-event-go

For each slot in the incoming batch, future-event-go checks if a FutureEvent already exists for that scheduleID:

¿Existed FutureEvent for this scheduleID?
│
├── YES (already existed)
│   ├── open NOT_SENT + openTime now in past → neutralize (mark SENT)
│   ├── close NOT_SENT + closeTime now in past → neutralize (mark SENT)
│   ├── close NOT_SENT + closeTime still future → update triggerTime if changed
│   └── already SENT → no action
│
└── NO (new slot)
    ├── open past + close past → ignore (no longer relevant)
    ├── open past + close future → ONE Kinesis event (OPERATING) + create close NOT_SENT
    ├── open future → create open NOT_SENT
    └── close future → create close NOT_SENT

Additionally, for FutureEvents that exist in the DB but are NOT in the incoming batch (slot was removed): → neutralize (mark SENT). This handles orphan cleanup.

Key insight: The only case that needs an immediate Kinesis event is: new slot + openTime in past + closeTime in future (entity should be operating right now). This is a single, precise event — not a broadcast of all past slots.

3.3 Changes Required

  1. New Lambda consuming from Schedule Changes Stream (simple: fetch + filter + forward)
  2. future-event-go: new "full sync" processor with the decision tree above
  3. future-event-go: add Kinesis client for operating status stream (for the one immediate-update case)
  4. FutureEvents schema: add scheduleID field

3.4 Edge Cases Analysis

Case 1: New rule creates 4 actual slots for today (it's 1:00 PM)

Slots: E1 (7-9AM), E2 (9AM-5PM), E3 (5-9PM), E4 (9-11PM)

New lambda sends batch to SQS. future-event-go receives it, reads existing FutureEvents (none exist):

  • E1 (7-9AM): new + both past → ignore
  • E2 (9AM-5PM): new + open past + close future + currently in range → ONE Kinesis OPERATING + create close NOT_SENT for 5PM
  • E3 (5-9PM): new + both future → create 2 FutureEvents NOT_SENT
  • E4 (9-11PM): new + both future → create 2 FutureEvents NOT_SENT

Result: 1 Kinesis event, 5 FutureEvents. Clean. ✅

Case 1b: 5 slots, all moved to past except last one's close

Slots 1-4 have both open and close in the past. Slot 5 has open in past, close in future.

  • Slots 1-4: if FutureEvents existed NOT_SENT → neutralize. If new → ignore (all past).
  • Slot 5: if new + open past + close future → ONE Kinesis OPERATING + create close NOT_SENT. If existed → neutralize old open, update close if changed.

Result: At most 1 Kinesis event. Precise. ✅

Case 2: Rule modified — E4 removed, E3 extended to 11PM

New lambda fetches via GET v2 → gets E1, E2, E3-new. Sends batch.

future-event-go reads existing FutureEvents, compares:

  • E1, E2: already SENT → no action
  • E3: existing NOT_SENT → update closeTime to 11PM
  • E4: existing NOT_SENT but NOT in incoming batch → neutralize (mark SENT)

Result: No orphans. E4 cleaned up. ✅

Case 3: User manually edits an engine actual

Manual save → existing OperatingStatusTrackSchedule flow with sourceID. Engine marks schedule as definitive: true → no more regenerations for that slot.

future-event-go's full sync processor only manages FutureEvents with engine scheduleIDs. Manual FutureEvents (with sourceID) are untouched.

Next time the batch arrives without the definitive slot → future-event-go sees existing engine FutureEvents not in batch → neutralizes them. ✅

Case 4: Engine regenerates but actuals don't change

future-event-go compares: all scheduleIDs match, all triggerTimes match → no writes, no Kinesis. ✅

Case 5: Slot moved from future to present

Existing FutureEvent open NOT_SENT at 5PM. Incoming: openTime 10AM (past), closeTime 9PM (future).

  • Old open FutureEvent → neutralize
  • New slot is in range now → ONE Kinesis OPERATING
  • Close FutureEvent → update to 9PM

✅

Case 6: Slot moved from present to all-future

Existing: open SENT, close NOT_SENT 10PM. Incoming: open 6PM, close 11PM (both future).

  • Old close (10PM) → neutralize
  • New open (6PM) → create NOT_SENT
  • New close (11PM) → create NOT_SENT
  • Entity was OPERATING but slot moved away → future-event-go detects this and sends Kinesis CLOSED

✅

4. Proposal B: Extend generate-schedules Lambda

4.1 Key Insight

generate-schedules is the only point in the system where both the old and new state are available simultaneously:

  • It reads the deprecated schedule from ScheduleCache (the old state)
  • It generates the new schedule via the resolver (the new state)
  • It writes the new schedule to ScheduleCache

This makes it the natural place to generate the batch message for future-event-go — no GET v2 needed, no guessing about what changed, no latency from HTTP calls.

4.2 Architecture

Deprecated Schedules Stream (Kinesis)
    → generate-schedules Lambda (existing)
        → Regenerates actuals (already does this)
        → Writes to ScheduleCache (already does this)
        → NEW: If scheduleType == ACTUAL:
            Build batch message with current slot list
            Send single SQS message to future-event-go
        → Publishes Schedule Change Event (already does this)

SQS → future-event-go
    → Same "full sync" processor as Proposal A
      (same decision tree, same logic)

4.3 Changes Required

  1. generate-schedules: after writing actuals to cache, build and send batch SQS message with current slot list
  2. future-event-go: same new "full sync" processor as Proposal A
  3. future-event-go: add Kinesis client for operating status stream
  4. FutureEvents schema: add scheduleID field

4.4 Edge Cases Analysis

All edge cases are handled identically to Proposal A (same future-event-go processor), with these additional advantages:

Case 1-6: Same as Proposal A

The future-event-go processor is the same. All cases resolve the same way. ✅

Case 7: Engine regenerates but actuals don't change

generate-schedules can compare new vs deprecated before sending SQS. If identical → skip SQS entirely. Zero unnecessary messages. ✅

(In Proposal A, the new lambda always fires because it receives the Schedule Change Event regardless.)

Case 8: GET v2 staleness (Proposal A problem)

Not applicable. generate-schedules already has the schedule data in memory — no HTTP call needed. ✅

Case 9: Manual override of engine actual

When a user edits an engine actual, it gets saved in publishedSchedules and the engine marks it definitive: true. On next regeneration, generate-schedules does not include the definitive slot in the batch → future-event-go sees existing engine FutureEvents not in batch → neutralizes them. Clean separation. ✅

Case 10: Cache deprecated without regeneration (error)

If generate-schedules fails, no batch is sent. Existing FutureEvents remain. When retry succeeds, batch is sent and FutureEvents are corrected. Eventual consistency. ⚠️ (acceptable)

Case 11: generate-schedules writes cache but fails sending SQS

Schedule updated in cache but FutureEvents not updated. Operating status based on old FutureEvents until next successful regeneration. ⚠️ (acceptable, same pattern as rest of system)

4.5 Advantages Over Proposal A

6. Shared Requirement: future-event-go Full Sync Processor

Both proposals require the same new processor in future-event-go. The difference is only in who sends the batch message (new lambda vs generate-schedules).

6.1 Batch Message Format

{
  "entityId": "ObjectID",
  "businessDate": "2024-08-01",
  "eventType": "SCHEDULE",
  "slots": [
    {
      "scheduleID": "hash1_actual",
      "openTime": "2024-08-01T09:00:00Z",
      "closeTime": "2024-08-01T17:00:00Z",
      "scheduleEvent": "Standard Operation",
      "closedReason": ""
    },
    ...
  ]
}

6.2 Processor Logic

  1. Read all existing FutureEvents NOT_SENT for this entity + businessDate (filtered by engine scheduleIDs only)
  2. Build maps: existing by scheduleID+isOpen, incoming by scheduleID
  3. For each incoming slot: apply decision tree (see section 3.2)
  4. For each existing FutureEvent NOT in incoming: neutralize (mark SENT) — this handles removed slots

7. Recommendation

Both proposals are viable now that the core logic lives in future-event-go's full sync processor. The choice comes down to:

  • Proposal A if you prefer not to modify generate-schedules and accept the GET v2 dependency and cascade handling complexity.
  • Proposal B if you prefer no new infrastructure, no HTTP dependency, and the ability to skip unchanged regenerations — at the cost of adding SQS responsibility to generate-schedules.

Both require the same changes to future-event-go and the FutureEvents schema.
